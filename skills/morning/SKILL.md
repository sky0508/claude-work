# /morning — 朝のブリーフィング v3

## 概要

コンテキストを自動読み込みし、カレンダー・Gmail・バックログを統合して今日の行動計画を設計する。
**オンボーディング質問は一切しない。すでに文脈は把握している前提で即座に動く。**

### バックログ優先の原則

```
【正のタスクデータ】
  company_jva/secretary/backlog.md    ← JVAタスクの唯一の正
  company_personal/tasks/backlog.md   ← 個人タスクの唯一の正

前日 /evening が更新済みのため、バックログは常に最新状態。
company_jva/secretary/todos/YYYY-MM-DD.md は記録用（参照のみ）。
```

---

## 実行フロー

### Step 1: コンテキスト自動読み込み（サイレント実行）

以下を順番に読み込む。ユーザーへの報告は不要。

```
company_personal/context/ai-preferences.md
company_personal/context/owner-profile.md
company_personal/tasks/recurring.md
company_personal/tasks/backlog.md          → 個人Q1/Q2タスクを抽出（正のデータ）
company_jva/secretary/backlog.md           → JVA Q1/Q2タスクを抽出（正のデータ）
company_jva/secretary/todos/ の最新ファイル → 前日の記録・夕方ログを確認
company_jva/kpi/db/weekly.md               → 直近KPIを確認
```

---

### Step 2: Google Calendar 取得（MCP）

MCPツール `google-calendar` を使って **今日の予定を取得**する。

```
取得範囲: 今日 00:00 〜 23:59
出力: 時刻・タイトル・場所（あれば）の一覧
```

**確認すること:**
- 「確定しているが準備が必要な予定」があるか（→ 準備タスクをQ1に追加）
- 空き時間ブロックの把握（タスク配置の根拠にする）

> ※ カレンダーにタスクは追加しない。予定の確認のみ。

---

### Step 3: Gmail 確認（MCP）

MCPツール `gmail` を使って **重要な未返信メールを抽出**する。

```
条件: 受信から3日以上経過 かつ 重要度が高い（星付き or 返信待ち）
除外: プロモーション・通知系・ニュースレター
出力: 件名・送信者・受信日・一言サマリー（最大5件）
```

---

### Step 4: スプレッドシート確認

**学生アプリケーション・企業アプリケーションの新着チェック。**

Google Sheetsへのアクセスがある場合はMCPで取得。ない場合は以下を出力:
```
⚠ スプレッドシート確認が必要: 学生・企業の新着申し込みを手動で確認してください。
```

---

### Step 5: 今日のタスク生成

以下の優先順位ロジックで今日のタスクを選定する:

```
【優先順位ロジック】
1. Q1（緊急×重要）→ 全件 Must に入れる
2. カレンダーに準備が必要な予定 → Q1扱いで Must に追加
3. 今日が締め切りのQ2タスク → Must に追加
4. 空き時間がある場合: Q2タスクを Should に最大2件追加
5. Q3（緊急×非重要）→ 定期タスクとして表示
6. 今日が該当するrecurringタスク → 定期タスクに追加

Must は最大4件に絞る。多すぎると機能しない。
```

**`tasks/today.md` を以下のフォーマットで生成（既存なら上書き）:**

```markdown
# 今日のタスク: YYYY-MM-DD（曜日）

---

## 📅 今日のカレンダー

- HH:MM [タイトル]（場所）
- HH:MM [タイトル]
...
空き時間: XX:XX〜YY:YY、ZZ:ZZ〜WW:WW

---

## 📧 未返信・要対応メール

- [件名] — [送信者] (X日前) ← 一言サマリー
...
（なければ「なし」）

---

## 📱 新着アプリケーション

- 学生: X件 / 企業: Y件（要確認）
（なければ「新着なし」）

---

## ✅ 今日やること

### 🔴 Must（緊急 × 重要）

- [ ] [タスク名] | [象限] | [理由/期限]
...（最大4件）

### 🟡 Should（余裕があれば）

- [ ] [タスク名] | Q2
...（最大2件）

### 🔄 定期タスク

- [ ] [タスク名]（定期: 毎日/毎週）

---

## 📊 KPIアラート

[直近KPIで目標未達・悪化傾向のものを1〜2件のみ]
（問題なければ「問題なし」）

---
*生成: /morning YYYY-MM-DD*
```

## 他スキルとの連携

| 場面 | 使うスキル |
|------|----------|
| 朝の定型ブリーフィング | `/morning`（このスキル）|
| 日中のタスク追加・相談・進捗確認 | `/secretary` |
| 夜の締め・進捗記録 | `/evening` |
| コード/深い作業のセッション終了 | `/fin_s` |

> `/morning` はブリーフィングを出力したら終了。その後のタスク管理は `/secretary` が担当。

---

## 重要ルール

1. **オンボーディング質問は絶対にしない** — owner-profile.md を読んで文脈は把握済み
2. **カレンダーにタスクを追加しない** — 予定の確認と空き時間把握のみ
3. **Mustは最大4件** — 絞り込みが仕事
4. **KPIアラートは悪化傾向のものだけ** — 毎朝全部出さない
5. **tasks/today.md は毎朝上書き** — 前日分は archive/ に退避してから上書き
6. **バックログを信頼する** — /evening が前日に更新済み。todos/ の持ち越しより backlog を優先
7. **タスクのダブりを出さない** — backlog にあるタスクを todos/ でも確認し、重複しない

---

## アーカイブ処理

`tasks/today.md` を上書きする前に、前日分が残っている場合:
```
tasks/archive/YYYY-MM-DD.md にコピーしてから today.md を上書き
```
