## 作業スタイル（判明分）

- 複数の役割を同時並行しているため、タスクが散らかりやすい
- 2人体制での運営が多い → リソース制約を常に意識している

## コミュニケーション好み

- 長い説明よりも、構造化された短い情報を好む傾向（初回セッションの会話から判断）
- 「まずやってみせて」派 → 理論より実践・実行重視
- 日本語で会話（英語混じりでも問題なし）

## 優先順位のつけ方

- ボトルネック解消優先（マッチング0件 → 原因特定 → 打ち手 の思考順序）
- 即効性のある打ち手と中期的な構造改善を並行して考えたい

### 2026-03-12 追記

- **次点**: 締切・期限が近いもの（やらざるを得ないので先に片付ける）

## 得意なこと・苦手なこと

- **得意**: 技術的なデバッグ・エラー解析（OAuth認証エラーを自分で調べて解決するレベル）
- **得意**: ツール・環境の整備に積極的（MCP連携・スキル修正を自ら進める）
- **苦手・要サポート**: CLIツール（tmux等）は初学者。「ペインとは何か」「Tabキーの場所」など基礎から丁寧に説明する必要あり（2026-03-20）

## AIに慣れてきた度合い

- Claude Codeを使い始め、/company でのファイルベース管理を初めて使っている
- スキルやコマンドの仕組みに興味がある（/fin_s を自分で要求した）

## 意思決定の境界線（2026-03-11 確定）

| 自分で決める | AIに任せる |
|------------|---------|
| 体験設計（何のために・どんな体験をしたいか） | 裏側の仕組み構築・実装 |
| 目的・WHY・やりたい作業の定義 | コード・ツール・自動化の実装 |
| 情報管理の目的と使い方 | ファイル構造・フロー設計 |

**重要**: 仕組みはAIに任せるが、**理解はしたい**。実装後に必ず「何をどう作ったか」の簡単な説明を求める。

### 2026-03-15 追記

- **技術インフラ系**（Git設定・セットアップ系）も全委任でOK。ただし完了後に「何をしたか」を簡潔に説明すること。
- **技術的トラブルシューティング**（MCPエラー・認証設定等）は「一緒に進める」スタイルを好む。手順を見せながら、何が起きているかを都度説明しつつ進める。

### 2026-03-16 追記

- **Partnership系ドキュメント設計フロー（対話→アウトライン→Manus）**: 対話スタイルを維持。毎回対話でアウトラインを詰めてからManusに渡す方式を継続する。

### 2026-03-18 追記：Notion × Claude Code 使い分け方針（確定）

**参照記事**: @reina_Notionxx「Notion AIガチ勢がClaude Code導入で最初にやったコンテキスト設計」

| 場所 | 置くもの |
|------|--------|
| **Notion** | コンテキスト全振り（タスク・プロジェクト・ログ・ナレッジ） |
| **Claude Code（ローカル）** | CLAUDE.md（行動ルールのみ）・Skills・MEMORY.md（技術情報のみ） |
| **Claude Code** | 実行エンジン（コード・実装・自動化の手足） |

**理想の役割分担:**
- Notion AI = 普段の秘書・コンテキスト把握・業務の大半
- Claude Code = 重い実装・Notion外の仕事・Notion AIのスペックを受け取って実行

**今後の方向性:**
- ローカルの `company_jva/` `company_personal/` への情報蓄積は Notion に移行していく
- Claude側にはルール・スキルのみを残す設計に収束させる
- save/resume スキルで Notion ↔ Claude Code のコンテキスト引き継ぎを実現する
